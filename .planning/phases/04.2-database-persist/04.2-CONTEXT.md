# Phase 04.2: database persist - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrar a persistência do app de repositórios in-memory para banco SQLite via Prisma em modo big-bang, garantindo continuidade funcional dos fluxos atuais e removendo o modo memória do runtime principal.

</domain>

<decisions>
## Implementation Decisions

### Estratégia de migração
- Migração será **big-bang** (troca completa em uma fase).
- Não manter fallback in-memory no fluxo principal após migração.
- Todos os módulos com repositório atual em memória entram no escopo: contas, cartões, categorias, transações e agendamentos.
- Em caso de falha durante operações, comportamento deve ser fail-fast com erro explícito.

### Estratégia de testes
- Persistência de testes deve usar **SQLite real**.
- Não manter dependência de arquitetura in-memory para validação principal da fase.
- Ajustes de teste podem alterar setup/reset para cenário de banco real desde que mantenham cobertura funcional atual.

### Bootstrap mínimo de dados
- Não usar dados demo genéricos na inicialização cega.
- Dados básicos devem ser criados no **fluxo de cadastro do usuário/conta inicial**.
- Após cadastro inicial, sistema cria baseline mínimo: algumas categorias padrão, uma conta e um cartão.
- Fluxo mínimo de onboarding de conta é obrigatório para app ficar utilizável.

### Erros e feedback ao usuário
- Para qualquer erro de persistência/operação, mostrar snackbar de erro na UI.
- Estratégia UX preferida: impedir sucesso parcial silencioso e comunicar falha imediatamente.
- Mensagens podem iniciar simples nesta fase, mas cobertura de erro deve ser ampla (todo tipo de erro operacional).

### Claude's Discretion
- Definir desenho técnico exato do bootstrap inicial (gatilho e idempotência).
- Definir forma de isolamento de banco em testes (reset por suíte/caso) mantendo estabilidade e tempo aceitável.
- Definir mapeamento de erros de baixo nível (Prisma/SQLite) para mensagens de UI sem vazar detalhes técnicos.

</decisions>

<specifics>
## Specific Ideas

- Diretriz explícita do usuário: “big bang, não manter nada em memória, migra tudo”.
- Diretriz de testes: “com sqlite”.
- Diretriz de onboarding: ao criar conta do usuário, gerar base mínima com categorias, uma conta e um cartão.
- Diretriz de UX de erro: snackbar para todo tipo de erro.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `prisma/schema.prisma` já define modelos necessários para os domínios atuais.
- Estrutura de módulos por domínio (`repository/service/controller`) já estabilizada e reutilizável na migração.
- Snackbar global já existe em `src/components/ui/snackbar.tsx`, pronto para mapear feedback de erro.

### Established Patterns
- Runtime atual em `src/app/foundation/runtime.ts` instancia repositórios in-memory concretos; este é ponto crítico da troca para Prisma.
- Serviços e controllers já centralizam regras de negócio; repositório pode ser substituído preservando contratos de uso.
- Testes atuais assumem `clearAll()` em repositórios; setup terá de ser redefinido para banco real.

### Integration Points
- Repositórios a migrar: `accounts`, `cards`, `categories`, `transactions`, `scheduling`.
- Fluxos dependentes: cashflow, invoices, free-balance, schedules.
- Camada de cadastro inicial/onboarding deve garantir criação de baseline mínimo (categorias + conta + cartão) sem quebrar UX atual.

</code_context>
