# Phase 04.2: Database Persist - Research

**Date:** 2026-03-03
**Scope:** migração big-bang de persistência in-memory para persistência real com cobertura de erros e bootstrap mínimo.

## 1. Reality Check do Código Atual

- O produto atual é uma SPA Vite/React que instancia controllers e repositories diretamente no browser (`src/app/foundation/runtime.ts`).
- Repositórios atuais são arrays in-memory (`accountsStore`, `cardsStore`, `transactionsStore`, `scheduledInstancesStore`).
- Existe modelagem Prisma + migrations (`prisma/schema.prisma`), porém não conectada ao runtime principal.

## 2. Risco Estrutural Principal

Prisma Client não roda no browser (depende de runtime Node). Logo:
- não é viável trocar in-memory por Prisma mantendo arquitetura atual 100% client-side;
- para persistência real em SQLite, é necessário mover camada de acesso a dados para backend (Node) e expor API para frontend.

## 3. Direção Técnica Recomendada

### 3.1 Big-bang com backend mínimo

- Introduzir um servidor Node (BFF/API) para hospedar controllers/services/repositories persistentes.
- Frontend deixa de usar controllers locais e passa a consumir endpoints HTTP.
- Repositórios Prisma tornam-se fonte única de verdade (sem fallback in-memory no runtime principal).

### 3.2 Compatibilidade de domínio

- Manter contratos de domínio atuais (DTOs e validações) para minimizar regressão funcional.
- Adaptar controllers/services para async/await de ponta a ponta.

### 3.3 Bootstrap de dados mínimos

No onboarding (criação da conta inicial), criar automaticamente:
- categorias básicas padrão,
- uma conta base,
- um cartão base.

Sem dados demo aleatórios fora desse fluxo.

### 3.4 Erros

- Padronizar resposta de erro na API (code + message).
- UI converte falhas em snackbar de erro para toda operação crítica.

## 4. Test Strategy (SQLite real)

- Testes de módulo e integração devem usar SQLite real isolado por suíte.
- Reset previsível com utilitário de truncamento/transação por suite.
- E2E mantém cobertura de fluxos principais, agora exercitando camada API.

## 5. Impactos por Área

- **Runtime:** substituição de instâncias locais por client HTTP.
- **Repositories:** implementação Prisma para todos os módulos financeiros core.
- **Services/Controllers:** async e mapeamento de erros persistentes.
- **UI:** feedback de erro universal via snackbar + onboarding mínimo orientado a conta inicial.
- **Testing:** migração de `clearAll` in-memory para reset de SQLite real.

## 6. Sequenciamento Seguro

1. Infra backend + Prisma repositories + contrato de erro.
2. Migração de runtime/front para API + onboarding/bootstrap mínimo.
3. Conversão da suíte de testes para SQLite real + hardening/regressão.

## 7. Out of Scope de 04.2

- Multiusuário completo com auth/roles robustos.
- Open Finance/import automático.
- Observabilidade avançada além de logging básico + snackbar de erro.

## 8. Readiness for Planning

- Existe clareza de boundary técnico (persistência real exige backend).
- Decisões do usuário estão lockadas (big-bang, sem memória, SQLite, bootstrap mínimo no cadastro, snackbar para erros).
- Fase pronta para quebrar em planos executáveis.
