<interfaces>
New controller contract to create in this plan:
```typescript
createCreditCardAdjustment(input: {
  householdId: string;
  cardId: string;
  realInvoiceTotal: string;
  dueMonth: string;
  occurredAt: string;
}): {
  previousInvoiceTotal: string;
  realInvoiceTotal: string;
  difference: string;
  transaction: TransactionRecord;
}
```

Adjustment transaction rule:
```typescript
// difference = realInvoiceTotal - previousInvoiceTotal
kind: "EXPENSE"
description: "REAJUSTE"
amount: difference.toFixed(2) // signed: positive increases invoice, negative decreases invoice
creditCardId: cardId
accountId: null
invoiceMonthKey: dueMonth
invoiceDueDate: `${dueMonth}-${card.dueDay}T00:00:00.000Z`
settlementStatus: null
transferGroupId: null
```

  <behavior>
    - `createCreditCardAdjustment` calcula `difference = realInvoiceTotal - previousInvoiceTotal`.
    - Quando `realInvoiceTotal` e maior que o total atual, cria `EXPENSE` `REAJUSTE` com amount positivo.
    - Quando `realInvoiceTotal` e menor que o total atual, cria `EXPENSE` `REAJUSTE` com amount negativo.
    - Categoria `Reajuste` e criada ou reutilizada por `householdId` com `normalized = "reajuste"`.
  </behavior>
  <action>Criar `CreditCardAdjustmentsService` com `InvoicesService`, `TransactionsRepository` e `CategoriesRepository`. Validar input com zod (`householdId`, `cardId`, `realInvoiceTotal`, `dueMonth`, `occurredAt`). Usar `InvoicesService.getCardInvoiceEntriesByDueMonth({ householdId, cardId, dueMonth })` para obter `previousInvoiceTotal` e validar ownership do cartao. Calcular a diferenca com `Decimal`. Criar/reusar categoria normalizada `reajuste`. Gravar uma transacao diretamente no `TransactionsRepository` com `kind: "EXPENSE"`, `description: "REAJUSTE"`, `amount` assinado, `creditCardId`, `invoiceMonthKey: dueMonth`, `invoiceDueDate` derivado do `dueDay` do detalhe da fatura, e sem conta/settlement/transfer. Nao implementar tratamento especial para diferenca zero; isso pertence a Phase 26.</action>
